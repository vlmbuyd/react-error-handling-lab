## 오늘 내가 스스로 깨달은 것

### 왜 class인가 (복습) — 함수는 pop돼 사라지고, 인스턴스는 React가 쥐고 있다
- 이끈 질문: "함수도 객체잖아 — 그럼 갈림길은 '객체냐'가 아니라 뭘까? React가 손에 쥐고 있는 쪽은?"
- 본 코드:
  ```jsx
  UserProfileV1()        // 실행 → JSX 반환 → 끝
  new ErrorBoundaryV1()  // React가 instance를 만들어 들고 있음
  ```
- 내 말 (원문 그대로):
  > 함수 즉 컴포넌트는 실행된 후 결과만 받고 함수 자체는 pop돼서 종료될 것 같아.
  > 그렇다면 클래스의 인스턴스는 React가 잡고 있는건가?
- **다듬은 정리**: 함수 컴포넌트는 실행 후 결과(JSX)만 남기고 함수 자체는 콜스택에서 빠진다. class 인스턴스는 React가 참조로 들고 있어 나중에 다시 부를 수 있다. 갈림길은 "객체냐"가 아니라 "누가 그걸 가리키고 있느냐"다.
- 맥락: 지난 세션 "왜 class" 복습. `ErrorBoundaryV1`/`UserProfileV1`.
- **점검**: 정확함. "힙에 있냐"에서 "누가 참조하느냐"로 기준을 옮긴 것이 핵심.
- **다음 한 걸음**: 그 인스턴스를 React가 들고 있어서, 자식이 터지면 무슨 메서드를 부를 수 있나? (→ 아래)

### 에러 흐름: getDerivedStateFromError 리턴값을 React가 state에 병합 → 재렌더 → fallback
- 이끈 질문: "`return { error }`만 하는데 어떻게 state가 바뀌나? render는 필요할 때마다 다시 불리나?"
- 본 코드:
  ```jsx
  static getDerivedStateFromError(error) { return { error }; }
  render() { if (this.state.error) return /* fallback */; }
  ```
- 내 말 (원문 그대로):
  > React의 메커니즘에 의해 해당 리턴된 에러값이 state를 업데이트시켜서 다음 render 메서드 호출 시 이를 참조할 수 있어. 그래서 fallback으로 넘겨줄 것 같아.
- **다듬은 정리**: `getDerivedStateFromError`는 직접 state를 저장하지 않고, 리턴한 객체를 React가 받아 state에 병합한다(`setState`와 같은 방식). state가 바뀌면 React가 같은 인스턴스의 `render()`를 다시 불러 fallback을 그린다.
- 맥락: `ErrorBoundaryV1.tsx` 21·32줄.
- **점검**: 정확함. "리턴값을 React가 처리한다"를 스스로 짚음. (사실 보강: render는 첫 마운트 + state/props 변경 시마다 다시 호출됨.)
- **다음 한 걸음**: 그럼 비동기로 잡은 에러는 어떻게 이 render 경로까지 올려보내나? (→ 12줄 트릭)

### 12줄을 지우면 — "영원한 로딩" 거짓 화면
- 이끈 질문: "`if (asyncError) throw asyncError`를 지우고 `?fail=true`면 화면에 뭐가 보일까?"
- 본 코드:
  ```jsx
  if (asyncError) throw asyncError;          // 지웠다고 가정
  useEffect(() => { fetchUser().then(setUser).catch(setAsyncError); }, []);
  if (!user) return <LoadingSkeleton />;
  ```
- 내 말 (원문 그대로):
  > 해당 줄을 지워버리면 로딩만 계속 돼. 왜냐하면 .then()체인이 실행되지 않기 때문에 user가 null일거고 그렇기 때문에 계속 로딩만 뜨는거야
- **다듬은 정리**: 거부되면 `.then(setUser)`를 타지 않아 `user`가 끝까지 null → `LoadingSkeleton`만 영원히 뜬다. 에러는 `.catch`로 잡혀 state에 들어와 있지만 화면엔 전달되지 않는다.
- 맥락: v1 [버티는 줄] 실험, `UserProfileV1.tsx` 12줄.
- **점검**: 정확함. 직접 돌려 확인.
- **다음 한 걸음**: 에러를 손에 쥐고도 로딩을 보여주는 게 왜 문제인가?

### 거짓 화면이 위험한 이유 — 상태와 화면이 어긋난다
- 이끈 질문: "에러를 state에 들고 있는데 화면은 영원히 로딩이야. 사용자 입장에서 뭐가 문제일까?"
- 내 말 (원문 그대로):

- **다듬은 정리**: 실제 상태(에러)와 화면(로딩)이 어긋나면 사용자는 데이터를 불러오는 중이라고 잘못 믿는다. 잘못된 정보를 전달하는 화면이다.
- 맥락: v0 "반쪽짜리 거짓 UI" 통찰과 같은 결.
- **점검**: 정확함.

### .catch 안에서 throw하면 ErrorBoundary에 안 닿는다 — Promise 체인 안에 갇힌다
- 이끈 질문: "state에 담긴 에러를 ErrorBoundary가 잡게 하려면, 왜 `.catch` 안(19줄)에서 throw하면 안 될까?"
- 내 말 (원문 그대로):
  > catch 안에서 throw를 하면 errorBoundary에 닿는 것이 아니라 Promise 체인 내부에 갖혀버려서 그런가?
- **다듬은 정리**: `.catch` 콜백은 render가 끝나 pop된 뒤 마이크로태스크에서 실행되므로, 그 안에서 throw해도 React의 render 콜스택 위가 아니다. ErrorBoundary는 render 경로의 throw만 잡으므로 닿지 않는다.
- 맥락: v0의 "render 경로 vs 그 외" 기준을 v1에 적용.
- **점검**: 방향 정확함("Promise 체인 내부에 갇힌다"). render 콜스택이 이미 빠진 뒤라는 타이밍이 근거.
- **다음 한 걸음**: 그럼 "렌더 중 throw"는 12줄에서 어떻게 만들어지나?

### 12줄 트릭의 메커니즘 — setState가 재렌더를 부르고, 그 재렌더 도중 throw
- 이끈 질문: "`setAsyncError`를 부르면 React는 뭘 하지? 그때 12줄을 다시 만나 throw하면, 이 throw는 어느 경로 위에 있나?"
- 본 코드:
  ```jsx
  .catch(setAsyncError);            // 19줄: 에러를 state에 담음
  if (asyncError) throw asyncError; // 12줄: 재렌더 때 여기서 throw
  ```
- 내 말 (원문 그대로):
  > setAsyncError를 호출하면 상태가 업데이트 된거니까 React는 해당 컴포넌트를 리렌더링하겠지. throw asyncError에서 에러가 터지면 현재 렌더링 중인 컴포넌트 콜스택 위야. 그래서 ErrorBoundary에게 에러가 전달될 수 있는거지
- **다듬은 정리**: `.catch`가 잡은 에러를 `setAsyncError`로 state에 담으면 React가 재렌더한다. 그 재렌더 도중 12줄에서 throw가 나면 이번엔 render 콜스택 위라서 ErrorBoundary가 가로챈다. 비동기 에러를 render 경로로 갈아끼우는 것.
- 맥락: `UserProfileV1.tsx` 12·19줄.
- **점검**: 정확함. setState→재렌더→렌더 중 throw 순서를 콜스택 위치로 정확히 연결.
- **다음 한 걸음**: 그럼 매 컴포넌트마다 이 두 줄을 반복하는 번거로움은? (→ v2 `showBoundary`)

### ErrorBoundary가 잡는 기준 = "render 도중"(초기든 재렌더든), 재렌더는 이 예제의 성질일 뿐
- 이끈 질문: "fetch 없이 첫 렌더부터 throw하는 컴포넌트도 잡힐까? 기준이 '재렌더 중'일까 '렌더 중'일까?"
- 내 말 (원문 그대로):
  > ErrorBoundary가 에러를 잡는 기준은 렌더 도중인 것 같아. 어느 렌더 시점인지는 상황에 따라 다른거고.
  > 이 예제 자체가 비동기 데이터에 의존하기 때문에 해당 데이터를 업데이트하고 리렌더링되는 도중 에러를 던지면서 잡을 수밖에 없었던거지
- **다듬은 정리**: ErrorBoundary는 초기 렌더든 재렌더든 "render 도중"의 throw를 잡는다. 우리 예제가 하필 재렌더에서 터지는 건 ErrorBoundary의 성질이 아니라, 이 예제가 비동기 데이터에 의존하기 때문이다.
- 맥락: `Boom` 반례(첫 렌더부터 throw)로 일반화 점검.
- **점검**: 정확함. "ErrorBoundary 성질"과 "예제 성질"을 정확히 분리.

### 왜 getDerivedStateFromError는 static이고 componentDidCatch는 인스턴스 메서드인가
- 이끈 질문: "왜 static일까? render 단계의 순수성과 무슨 관계?"
- 본 코드:
  ```jsx
  static getDerivedStateFromError(error) { return { error }; }   // render 단계, 순수
  componentDidCatch(error, info) { console.error(...); }         // commit 단계, 로깅
  ```
- 내 말 (원문 그대로):
  > getDerivedStateFromError를 static으로 둔 이유는 순수해야 하는 렌더링 단계에 호출되는 해당 메서드가 this.state를 건들지 못하고 에러를 받아서 반환만 하도록 순수함수로 만들기 위함이야. componentDidCatch는 커밋 페이즈 ... 순수 로깅하는 함수이므로 state에 접근할 수 있는거지 (static 아님)
- **다듬은 정리**: `getDerivedStateFromError`는 render 단계에서 불리므로 `static`으로 `this` 접근을 막아 "에러→새 state"만 하는 순수 함수로 강제한다. `componentDidCatch`는 commit 단계에서 불려 로깅 같은 부수효과를 맡으므로 인스턴스 메서드다.
- 맥락: `ErrorBoundaryV1.tsx` 20·25줄.
- **점검**: 핵심 정확함. 사실 한 가지 교정: `componentDidCatch`는 commit 단계이지만 "브라우저 페인트 후"가 아니라 보통 페인트 직전에 동기 실행됨(페인트 후는 `useEffect` 쪽).
