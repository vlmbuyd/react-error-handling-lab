# 2026-06-15 — try/catch·`.catch` 에러 채널 정리와 시나리오 실습, ErrorBoundary 진입

> **오늘 한 줄 요약**: v0 마무리 — 두 채널(try/catch·`.catch`)의 경계와 "흰 화면 기준 = render 경로 vs event 경로"를 자기 말로 닫음.
> 이어서 v1(ErrorBoundary) 진입 — render는 자식을 실행하지 않고 설명서만 반환(①→②)한다는 걸 직접 확인하고,
> **"왜 아직 class인가"(함수는 pop돼 사라지고 class 인스턴스는 남아 React가 다시 부른다)까지 자기 말로 도달.**

- 다룬 단계: v0(마무리) → v1(진입)
- 다룬 시나리오: 시나리오 1(서버에러/`.catch`) 복습 · 시나리오 3(이벤트 핸들러 throw) · v1 ErrorBoundary 구조

## 오늘 내가 스스로 깨달은 것

### `.catch`가 실행됐다 = 거부에 핸들러가 있었다 = uncaught가 아니다

- 이끈 질문: "거부가 일어났을 때 `.catch` 콜백이 실행됐다는 사실은 — 처리해줄 핸들러가 있었다는 뜻일까요, 없었다는 뜻일까요? 그게 uncaught와 무슨 관계죠?"
- 본 코드 (발췌):
  ```jsx
  fetchUser().then(setUser).catch(setUserError); // 30줄 — 실행되면 "유저 에러" 표시
  ```
- 내 말 (원문 그대로):
  > .catch 콜백 내부의 onRejection 핸들러가 처리해줬겠지.
  > 에러를 잡았으니까 uncaught error는 안뜨겠지 아마?
- **다듬은 정리**: 화면에 "유저 에러"가 떴다는 건 `.catch(setUserError)`가 실제로 실행돼 state를 세팅했다는 증거다. `.catch` 콜백이 실행됐다는 것 자체가 그 거부에 처리 핸들러가 있었다는 뜻이고, 핸들러가 있었으니 uncaught가 아니다. 지난 세션 끝의 혼동("결국 uncaught 아닌가?")은 여기서 풀림 — 거부는 `.catch`가 깨끗이 잡았다.
- 맥락: `UserProfileV0.tsx` 30줄, 시나리오 1 복습.
- **이해도 점검**: 정확함. "`.catch` 실행됨 = 핸들러 있었음 = uncaught 아님"의 연결을 스스로 짚음.
- **더 나아가려면**: 그럼 `.catch`가 **없었다면**? (06-11 콘솔 예제처럼 샌다 — A/B 구분)

### 바깥 try/catch는 데드코드 — 동기 실행 후 스택에서 pop되니까

- 이끈 질문: "같은 `fetchUser()`를 감싼 바깥 try/catch는, 시나리오 1에서 한 번이라도 실행될까요? 안 된다면 왜죠?"
- 본 코드 (발췌):
  ```jsx
  try {
    fetchUser().then(setUser).catch(setUserError)  // 거부는 .catch가 잡음
  } catch (e) {
    setUserError(e as Error)   // 32~35줄 — 영영 안 들어옴
  }
  ```
- 내 말 (원문 그대로):
  > fetchUser와 여기에 체이닝된 코드들이 동기적으로 실행된 후에는 에러가 아직 뜨지 않은 상황이고 (fetchUser가 데이터를 아직 못 받아온 상황이므로) 그 상황에서는 try/catch 블록이 종료되어서 콜스택에서 pop되지. 그리고 데이터를 받아온 후에 에러가 뜬다면 체이닝되어 있는 catch()의 핸들러가 해당 에러를 잡겠지. 결론적으로는 catch(){} 블록은 실행될 일이 없어
- **다듬은 정리**: try 블록은 `fetchUser()` 호출을 동기적으로 통과한 직후 끝나 콜스택에서 pop된다. 그 시점엔 아직 거부가 일어나지 않았다. 나중에 거부가 도착하면 그땐 바깥 try/catch는 이미 사라졌고, 체인에 달린 `.catch`가 그 거부를 받는다. 그래서 바깥 `catch (e) {}`는 실행될 일이 없는 데드코드다.
- 맥락: `UserProfileV0.tsx` 24~35줄, 같은 fetch를 두 겹(try/catch + `.catch`)으로 감싼 구조.
- **이해도 점검**: 정확함. "동기 실행 후 try 블록 pop → 거부는 나중에 도착 → `.catch`가 받음"을 콜스택 타이밍으로 정밀하게 설명함.
- **더 나아가려면**: 그럼 "try/catch가 잡는 것"과 "`.catch`가 잡는 것"의 경계를 각각 한 문장으로?

### v0 닫는 문장 — try/catch와 `.catch`는 서로 다른 채널, 따로 떼야 한다

- 이끈 질문: "30줄 `.catch`는 비동기 거부를 잡았고 32줄 try/catch는 못 잡았어요. 둘을 한 묶음으로 부르는 게 맞을까요? `.catch`가 못 잡는 에러도 있을까요?"
- 본 코드 (발췌):
  ```jsx
  // 시나리오 2: 렌더 도중 throw — Promise 체인 '밖'
  <UserCard city={user.profile!.address.city} />  // .catch 시야 밖 → 흰 화면
  ```
- 내 말 (원문 그대로):

  > try/catch는 동기 에러는 잡지만, 비동기 에러는 못잡는다. catch()는 비동기 에러와 비동기 에러 모두 잡을 수 있다.

  > 아 그러면 .catch()에러는 Promise 체인 내부에서 일어난 에러만 잡고 그 외의 에러는 잡지 못한다. 이게 정확하겠네?

- **다듬은 정리**: try/catch와 `.catch`는 같은 것이 아니라 서로 다른 채널이라 따로 떼야 한다. (1) **try/catch**(동기 문장)는 스택에 살아있는 동안의 동기 throw만 잡는다. (2) **`.catch`**(Promise 체인)는 자기 체인 **내부**에서 일어난 거부만 잡고, 체인 밖에서 던져진 에러(예: 시나리오 2의 렌더 throw)는 못 잡는다.
- 맥락: 시나리오 1·2를 하나의 기준으로 묶는 v0 consolidation.
- **이해도 점검**: 정확함. 처음엔 둘을 한 묶음으로 적었으나, 직접 "따로 떼어야 한다"고 교정하고 `.catch`의 경계("체인 내부만")까지 스스로 정밀화함.
- **더 나아가려면**: 그럼 체인 밖에서 터진 렌더 throw를 *잡을 수 있는 곳*은? (→ v1 ErrorBoundary)

### 시나리오 3 — 이벤트 핸들러 throw는 화면이 멀쩡하다 (들어낼 UI가 없으니까)

- 이끈 질문: "버튼을 클릭한 순간 화면은 이미 어떤 상태였나요? 그렇다면 React 입장에서 '들어내야 할 거짓말 UI'가 존재하나요?"
- 본 코드 (발췌):
  ```jsx
  <button onClick={() => { throw new Error('이벤트 핸들러 에러!'); }}>  // 66줄
  // 콜스택: onClick → executeDispatch → processDispatchQueue → dispatchEvent (event 경로)
  ```
- 내 말 (원문 그대로):

  > 정확히는 렌더 후 onClick 핸들러가 클릭되었을 때 뜨지 ... 즉 두 줄의 에러 시점이 달라

  > 일단 이 에러가 콘솔에만 뜨고 화면이 흰 화면으로 바뀌거나 하지는 않아

  > 화면은 이미 렌더가 완료 되어서 멀쩡히 떠 있어. 그렇기 때문에 클릭 시 에러만 던지는 현 상황에서는 굳이 드러낼 ui는 없을 것 같아 . 핸들러 클릭 시 비동기로 데이터를 불러오는 상황이면 또 모르겠지만

- **다듬은 정리**: 시나리오 2(렌더 중 throw)와 달리, 이벤트 핸들러 throw는 렌더가 끝난 뒤 클릭 시점에 일어난다. 실제 콜스택도 render 경로(`renderWithHooks`/`beginWork`)가 아니라 event 경로(`executeDispatch`/`dispatchEvent`)였다. 이때 화면은 이미 완성·커밋돼 멀쩡히 떠 있으므로 React가 들어낼 "반쪽짜리 거짓말 UI"가 없다. 그래서 콘솔에 uncaught 에러만 남고 화면은 그대로다.
- 맥락: 시나리오 3 실행 후 실제 콘솔 콜스택(event 경로) 관찰.
- **이해도 점검**: 정확함. "에러 시점이 다르다(렌더 후 클릭)" → "이미 완성된 화면이라 들어낼 UI가 없다"를 지난 세션의 '반쪽 UI = 거짓말' 통찰과 스스로 연결함.
- **더 나아가려면**: 덧붙인 단서("비동기로 데이터 불러오면 또 모르지")를 끝까지 따라가면? (→ 아래 항목)

### 흰 화면을 가르는 진짜 기준 — 동기/비동기가 아니라 render 경로 vs event 경로

- 이끈 질문: "흰 화면이냐 아니냐를 가르는 진짜 기준이 '동기냐 비동기냐'일까요, '어느 경로에서 throw가 났냐'일까요?"
- 본 코드/상황 (발췌):
  ```
  클릭 → fetch → setState → 다시 렌더 → 그 렌더 도중 throw
       (트리거는 event, 하지만 throw는 render 경로에서)
  ```
- 내 말 (원문 그대로):
  > 지금까지의 학습 내용을 바탕으로 유추해보자면 흰 화면이냐 아니냐를 가르는 기준은 어느 경로 (render vs event)에서 에러가 났냐일 것 같아
  > 핸들러 클릭 시 비동기로 데이터를 불러오는 상황에서 에러가 발생한다면 흰 화면이 뜰것 같아
  > 트리거 자체는 이벤트이지만 결국 데이터를 불러와서 "렌더"하는 과정을 거치기 때문에
- **다듬은 정리**: 흰 화면(트리 언마운트)을 가르는 기준은 동기/비동기가 아니라 throw가 **어느 경로**에서 났느냐다. 클릭이 트리거여도 그 결과(fetch → setState)가 다시 *렌더*를 거치다 터지면 render 경로 → 흰 화면. 이벤트 핸들러 안에서 직접 터지면 event 경로 → 화면 멀쩡. 세 시나리오가 이 하나의 기준으로 묶인다.
- 맥락: 시나리오 3 단서를 일반화해 도출한 v0 capstone.
- **이해도 점검**: 정확함. 직접 돌려보지 않은 케이스(클릭→fetch→재렌더 throw)를 기존 원리로 **유추**해 결론까지 도달한 점이 핵심. 기준을 "sync/async"에서 "render path/event path"로 옮긴 것이 정확한 일반화.
- **더 나아가려면**: render 경로의 throw를 잡는 그 "경계"는 어떻게 생겼나? (→ v1)

### (v1 진입) ErrorBoundary는 부모에 위치하고, `getDerivedStateFromError`는 React가 자동 호출한다

- 이끈 질문: "이 파일 어디에서 우리가 `getDerivedStateFromError`를 호출하나요? 못 찾으면 — 누가 부르죠?"
- 본 코드 (발췌):
  ```jsx
  // StageV1.tsx — 경계가 자식을 감쌈
  <ErrorBoundaryV1 fallback={...}><UserProfileV1 /></ErrorBoundaryV1>
  // ErrorBoundaryV1.tsx — 우리가 부르는 줄이 없다
  static getDerivedStateFromError(error) { return { error }; }
  ```
- 내 말 (원문 그대로):

  > UserProfileV1보다 부모 쪽에 위치하지 않을까? 공식문서를 참고했을 때 렌더링 도중의 에러를 잡을 수 있을 것 같아

  > 네 말대로 getDerivedStateFromError를 호출하는 줄이 없어. ... 예측해보자면 ... 아마 해당 함수를 정의해놓으면 React가 알아서 자체적으로 렌더링 도중 에러가 발생하면 해당 함수를 호출하여 실행해주는 식인건가?

- **다듬은 정리**: ErrorBoundary는 터지는 컴포넌트의 **부모**에 위치해 자식의 렌더 중 throw가 올라오다 잡힌다. `getDerivedStateFromError`는 우리가 호출하는 코드가 없다 — 정의만 해두면 자식이 렌더 중 throw할 때 **React가 알아서** 그 메서드를 부르고 리턴값을 다음 state로 삼는다.
- 맥락: `StageV1.tsx` 구조 + `ErrorBoundaryV1.tsx` 21줄.
- **이해도 점검**: "부모 위치" "React가 자동 호출"까지는 스스로 맞게 도달. 이어진 **"왜 class여야 하나"는 같은 세션에서 끝까지 풀어냄** → 아래 두 항목 참고.

### render는 자식을 실행하지 않는다 — "그려라"라는 설명서만 돌려주고 끝난다 (①→② 직접 확인)

- 이끈 질문: "`return this.props.children`은 자식을 *실제로 실행*해서 돌려주는 걸까요, 아니면 '자식을 그려라'라는 설명서(객체)만 돌려주고 끝나는 걸까요? `①`(경계 render)과 `②`(자식 render) 중 뭐가 먼저 찍힐까요?"
- 본 코드 (발췌):
  ```jsx
  // ErrorBoundaryV1.render()
  console.log('① 경계 render'); return this.props.children;
  // UserProfileV1 본문 맨 위
  console.log('② 자식 render');
  ```
- 내 말 (원문 그대로):
  > 1 -> 2 순서대로 찍혀.

  > return this.props.children은 자식을 실제로 실행하지는 않을 것 같아. 일반 컴포넌트의 children과 동일하게 동작할 것 같아
- **다듬은 정리**: `ErrorBoundaryV1.render()`는 `return this.props.children`을 만나면 자식을 실행하는 게 아니라 "여기에 자식을 그려라"라는 설명서(React 엘리먼트 객체)만 돌려주고 끝난다. 그래서 콘솔도 `①`(경계 render)이 먼저 찍혀 끝난 뒤에야 `②`(자식 render)가 찍힌다. 즉 자식이 throw하는 `②` 시점엔 경계의 render(`①`)는 이미 끝나 스택에서 빠진 뒤다.
- 맥락: `ErrorBoundaryV1`/`UserProfileV1`에 `console.log`를 넣어 렌더 순서를 직접 관찰.
- **이해도 점검**: 정확함. "render는 자식을 실행하지 않고 설명서만 반환한다 → 부모 render가 먼저 끝나고 자식이 나중에 실행된다"는 순서를 직접 실행으로 확인함. 이게 "왜 class"의 타이밍 근거가 됨.
- **더 나아가려면**: 그렇다면 ②에서 터질 때 ①(과 그 안의 hook)은 어디에 있나?

### 왜 ErrorBoundary는 class여야 하나 — 함수는 pop돼 사라지고, class 인스턴스는 남아 React가 다시 부른다

- 이끈 질문: "render가 이미 끝나 빠진 뒤 자식이 ②에서 throw하면, 함수 컴포넌트 안의 `useCatchError()` hook은 그걸 잡을 수 있을까요? 못 잡는다면 class는 hook이 못 가진 무엇을 가져서 잡을 수 있죠?"
- 본 코드 (발췌):
  ```jsx
  // class라서 인스턴스가 렌더 후에도 살아있고, 이름표 붙은 메서드를 가짐
  static getDerivedStateFromError(error) { return { error }; }
  render() { if (this.state.error) return this.props.fallback(...); }
  ```
- 내 말 (원문 그대로):
  > 함수 컴포넌트도 어쨋든 객체 아닌가?

  > ErrorBoundary가 함수형 컴포넌트 + hook 조합이었다면 렌더를 하고 콜스택에서 pop되어서 없어질 것이기 때문에 자식 컴포넌트에서 에러를 던져도 잡을 수 없어. 하지만 ErrorBoundary가 클래스로 이루어진 객체라면 렌더링된 후에도 살아있기 때문에 자식 컴포넌트에서 에러를 던져도 다시 돌아와서 잡아주고 다시 렌더링해줄 수 있다는거 아니야?

  > 그리고 이 클래스의 render 함수는 React에 의해 사전 예약되어 있는 키워드라서 React가 특별하게 인식할 수 있다는거고?
- **다듬은 정리**: ErrorBoundary가 함수 컴포넌트 + hook이었다면 렌더하고 콜스택에서 pop돼 사라지므로, 나중에(②) 자식이 throw해도 잡을 주체가 남아있지 않다. 반면 class 인스턴스는 렌더가 끝나도 살아있어서, 자식이 터지면 React가 다시 돌아와 그 인스턴스의 메서드를 호출해 에러를 잡고 다시 렌더링할 수 있다. React는 class에 `render`/`getDerivedStateFromError` 같은 약속된 이름의 메서드가 있는지를 보고 에러 경계로 인식한다.
- 맥락: ①→② 렌더 순서 + "render는 return하면 pop된다"를 합쳐 "왜 class"를 자기 말로 닫음 (핵심 질문 1번의 마지막 조각).
- **이해도 점검**: 핵심 정확함 — "함수+hook은 pop돼 사라져 못 잡음 / class 인스턴스는 살아남아 React가 다시 와서 잡고 재렌더". 두 가지 정밀화: (1) "함수도 객체 아닌가?" — 맞지만 갈림은 '객체냐'가 아니라 **'React가 나중에 이름으로 부를 메서드(전화번호)가 붙어 있냐'**다. 함수 컴포넌트는 그 번호를 안 남기고, class 인스턴스는 `getDerivedStateFromError`라는 번호를 남긴다. (2) "사전 예약된 키워드"는 JS 키워드가 아니라 **React가 찾는 약속된 이름(계약, 덕 타이핑)**이 정확.
- **더 나아가려면**: (다음 세션) class 인스턴스와 함수 컴포넌트는 *근본적으로* 뭐가 다르길래 하나는 pop돼 사라지고 하나는 메서드를 기억해 살아남나?
