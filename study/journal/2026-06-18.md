# 2026-06-18 — use → cache → query: 인프라와 정책의 분리

- 다룬 단계: use, cache, query
- 다룬 시나리오: 시나리오1(서버에러) 재확인 / cache 무한 suspend / query 캐시·재요청

## 오늘 내가 스스로 깨달은 것

### use가 흡수한 것 — 보일러플레이트
- 이끈 질문: "동작이 v3랑 똑같다면, use가 v3에서 바꾼 건 '동작'이 아니라 뭐예요? 뭐가 사라졌죠?"
- 본 코드:
  ```js
  const userPromise = fetchUser();      // 바깥에서 한 번
  const user = use(userPromise);        // read()의 switch 세 갈래가 한 줄로
  ```
- 내 말 (원문 그대로):
  > 직접 구현한 보일러 플레이트가 사라진거지.
  > status의 케이스에 따라 반환하거나 던지는 값도 캡슐화했고, 상태 선언 및 업데이트도 use 내부에 내장되어 있겠지

- **다듬은 정리**: use는 v3 `createResource`의 보일러플레이트를 통째로 흡수했다. status 분기(반환/던지기)와 상태 선언·업데이트가 use 내부로 들어갔다. 동작(throw로 부모에 위임)은 v3와 같고, 표현만 줄었다.
- 맥락: use 탭에서 v3 `createResource.ts`와 `UserProfileUse.tsx`를 비교하며.
- **점검**: 정확함. use가 흡수하지 못하고 남긴 것(promise를 바깥에서 만드는 일 = 참조 안정)이 다음 단계의 입구라는 것까지 이어서 짚음.
- **다음 한 걸음**: use가 남긴 그 한 줄(`const userPromise = ...`)을 컴포넌트 안으로 옮기면?

### cache — 안에서 만들면 무한 로딩
- 이끈 질문: "userPromise를 컴포넌트 안으로 옮기면 무슨 일이 날 것 같아요?"
- 본 코드:
  ```js
  const user = use(inline ? fetchUser() : stablePromise);
  // inline=true면 매 렌더 새 promise
  ```
- 내 말 (원문 그대로):
  > 컴포넌트 안으로 옮기면 v3와 같이 무한 로딩이 발생할 것 같아.
  > 컴포넌트가 호출될 때마다 Promise 값이 초기화되어서 렌더링 간의 상태를 유지하고 사용할 수 없는거지
  > 컴포넌트 호출 -> Promise pending -> fallback -> resolve -> 컴포넌트 호출 -> Promise pending
  > Promise 값이 유지가 되지 않는 것이 가장 큰 문제인 것 같아

- **다듬은 정리**: render 안에서 만든 promise는 매 렌더 새 객체다. use는 매번 새 pending을 throw하고, resolve되기 전에 다음 렌더가 또 새 promise를 만든다. 그래서 fallback이 영원히 안 사라진다. 네트워크 탭엔 요청이 계속 쌓인다.
- 맥락: cache 탭 "render 안에서 생성(깨짐)" 토글 + 네트워크 탭 관찰.
- **점검**: 흐름(호출→pending→fallback→...)은 정확. 한 가지 표현 정정 — "Promise 값이 초기화된다"가 아니라, 저장된 게 애초에 없고 매 렌더 새로 태어나 결과를 볼 새도 없이 또 태어나는 것.
- **다음 한 걸음**: 그럼 props(userId)를 보면서 무한 로딩도 안 나려면 무엇이 어디에 있어야 하나?

### 모듈 스코프 캐싱의 한계 — 두 가지 문제
- 이끈 질문: "이 두 개(props 못 봄 / 무한 suspend)를 동시에 만족시키려면, 무엇이 어디에 있어야 할까요?"
- 내 말 (원문 그대로):
  > 해당 Promise는 인메모리(힙)에 저장되어서 캐시가 될거야.
  > 하지만 문제가 있는데
  > - 동적 파라미터가 필요할 때 받을 수 없다는 것. 동적 파라미터를 받기 위해 컴포넌트 내부에서 호출하면 컴포넌트 호출 시마다 Promise가 초기화되면서 모듈 스코프의 캐싱 이점을 가져가지 못함 (무한 로딩 문제 발생)
  > - 동적 파라미터에 따라 여러 Promise가 생겼을 때 각 Promise를 구별할 수 있는 고유의 key 부재
  > 이 두가지 문제를 모두 풀수 있으려면
  > - 함수 내부에서 호출되어서 동적 파라미터를 받을 수 있고
  > - 여러개의 Promise 호출을 구분할 수 있는 key가 있어야 한다.
  > 그리고 이걸 라이브러리로 구현한 것이 useSuspenseQuery이다.

- **다듬은 정리**: 모듈 스코프 promise는 칸 1개·이름표 없는 캐시다. ① props로 다른 데이터를 받으려면 안에서 호출해야 하는데 그러면 무한 로딩, ② 여러 promise를 구별할 key가 없다. 둘을 풀려면 안에서 호출(파라미터) + key(구별)가 필요하고, 그걸 라이브러리로 구현한 게 useSuspenseQuery다.
- 맥락: use 탭의 `const userPromise = fetchUser()`를 두고.
- **점검**: 정확함. 두 문제를 분리해 짚고 해법도 두 조건으로 묶음. "인메모리(힙)에 저장돼 캐시가 된다"도 맞음 — 다만 lookup 가능한 캐시 자료구조가 아니라 변수가 객체 참조를 붙들고 있는 것.
- **다음 한 걸음**: 그 "사물함"은 어디에 살아야 재렌더에도 안 사라지나?

### 목록 카드 — 바깥 const 하나의 무용지물
- 이끈 질문: "멤버 카드 10개를 떠올리고, 바깥 const 하나로 박는 방식이 왜 무용지물이에요?"
- 내 말 (원문 그대로):
  > 바깥 const는 한 명만 담아서 10개 카드가 다 같은 사람으로 보임.

- **다듬은 정리**: `{ids.map(id => <UserCard userId={id} />)}`처럼 한 화면에 카드가 여러 개일 때, 바깥 const는 한 명만 담으므로 모든 카드가 같은 사람이 된다. 컴포넌트는 props로 다른 값을 꽂는 재사용 틀인데, 데이터를 바깥 const 하나에 박으면 그 틀과 어긋난다.
- 맥락: "한 화면에 userId가 다르게 들어오나?"라는 의문에 목록 렌더링 예를 깔고.
- **점검**: 정확함.
- **다음 한 걸음**: query는 카드 10개 각각에 다른 데이터를 어떻게 주나?

### 인프라 vs 정책
- 이끈 질문: "use가 안 하는 '낡았나/저장할까/재시도할까' 결정들을 누가 하고, 한 단어로 뭐라 부를까요?"
- 본 코드:
  ```js
  const { data: user } = useSuspenseQuery({ queryKey: ['user'], queryFn: fetchUser });
  // isLoading / isError 분기 없음
  ```
- 내 말 (원문 그대로):
  > 둘은 달라. (저장 vs 요청정책)
  > use는 Promise를 받아서 어떻게 처리할지만 알고 그 외의 정책들은 React가 하는 것 같아

- **다듬은 정리**: "값을 저장한다(캐시)"와 "언제 다시 요청한다(정책)"는 별개다. use는 던져진 promise를 처리하는 기계장치만 안다. 캐시·staleTime·retry 같은 결정은 그 위에 얹힌 정책이고, 이걸 맡는 건 React Query(라이브러리)다.
- 맥락: query 탭에서 fallback은 안 뜨는데(=캐시 적중) 네트워크 요청은 또 나가는 걸 보고. 기본값 staleTime:0의 stale-while-revalidate.
- **점검**: "둘은 다르다"는 정확. 단어 하나 정정 — 정책을 맡는 건 React가 아니라 React Query. use=React가 준 인프라, 정책=React Query라는 별도 라이브러리의 층.
- **다음 한 걸음**: useSuspenseQuery에 isLoading/isError 분기가 없는 건 인프라/정책 구분과 어떻게 이어지나?
