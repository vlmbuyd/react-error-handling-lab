## 오늘 내가 스스로 깨달은 것

### 💡 예제 1 — `.then`은 호출만 동기, 콜백은 비동기 (A → C → B)
- 이끈 질문: "`Promise.resolve().then(...)` 에서 then 안 콜백 B, then 다음 줄 C는 어떤 순서로 찍힐까?"
- 본 코드 (발췌):
  ```js
  console.log('A: 시작');
  Promise.resolve().then(() => console.log('B: then 안 콜백'));
  console.log('C: then 다음 줄');
  // 예측: A → ? → ?
  ```
- 내 말 (원문 그대로):
  > 예제 1- Promise의 체이닝된 then은 먼저 동기적으로 실행된 후 콜백만 예약한 후 내부 코드는 비동기적으로 실행하므로 A -> C -> B 순으로 찍힌다.
- **다듬은 정리**: `.then()` **메서드 호출 자체는 동기**다 — 호출되는 순간 콜백을 등록(예약)하고 새 Promise를 반환하며 즉시 끝난다. 등록된 **콜백 본문만** 나중에(현재 동기 코드가 콜스택을 다 비운 직후, 마이크로태스크 큐에서) 실행된다. 그래서 A(동기) → C(동기) → B(콜백) 순.
- 맥락: `Promise.resolve().then(() => log('B')); log('C')` 콘솔 순서 관찰.
- **이해도 점검**: 정확함. 한 가지만 또렷하게 — "then이 동기적으로 실행"이 아니라 "**then 호출은 동기, then에 넘긴 콜백은 마이크로태스크**"로 둘을 분리하면 완벽하다.
- **더 나아가려면**: `setTimeout(…, 0)`(매크로태스크)을 같이 섞으면 콜백 B와 어느 쪽이 먼저 찍힐까? (마이크로태스크 vs 매크로태스크 우선순위)

### 💡 예제 2 — 비동기 throw를 try/catch가 못 잡는 진짜 이유: 스택에 catch가 안 남아있음
- 이끈 질문: "동기 throw는 try/catch가 잡는데, `.then` 콜백 안의 throw는 같은 try/catch가 잡을까?"
- 본 코드 (발췌):
  ```js
  try {
    Promise.resolve().then(() => { throw new Error('비동기 throw'); });
  } catch (e) {
    console.log('잡음:', e.message);   // 이 줄이 찍힐까? → 안 찍힘
  }
  console.log('try 블록은 이미 끝났음');
  ```
- 내 말 (원문 그대로):
  > 예제 2- Promise.then()의 동기적 동작 -> 비동기적 동작 때문에 동기 동작 후 비동기 동작으로 에러를 잡을 때는 콜스택에 이미 에러를 잡아줄 try/catch문이 남아있지 않기 때문이다.
- **다듬은 정리**: `try/catch`는 **콜스택의 특정 프레임에 묶인** 안전망이다. 콜백이 `throw` 할 때쯤이면 그 try 블록은 이미 완료되어 **스택에서 pop**됐고, 콜백은 **완전히 새로운 콜스택**(마이크로태스크)에서 실행된다. 그 새 스택에는 try/catch가 없으니 잡을 사람이 없어 Unhandled Rejection으로 샌다.
- 맥락: `try { Promise.resolve().then(() => { throw … }) } catch(e){}` — catch 줄이 안 찍히는 걸 확인.
- **이해도 점검**: 정확함. 핵심("스택에 try/catch가 안 남아있다")을 정확히 짚었다.
- **더 나아가려면**: 그럼 이 비동기 throw를 **잡으려면** 어디에 핸들러를 둬야 하나? (예제 3의 `.catch`로 자연스럽게 연결)

### 💡 예제 3 — try/catch와 `.catch()`는 다른 채널: 바깥 catch는 이 거부에 대해선 죽은 코드
- 이끈 질문: "`try { Promise.reject().catch(...) } catch(e){}` 에서 바깥 try/catch와 `.catch` 중 어느 쪽이 거부를 잡을까?"
- 본 코드 (발췌):
  ```js
  try {
    Promise.reject(new Error('reject 발생!'))
      .catch((e) => console.log('.catch 채널이 잡음:', e.message));  // 이것만 찍힘
  } catch (e) {
    console.log('try/catch 채널이 잡음:', e.message);  // 안 찍힘
  }
  ```
- 내 말 (원문 그대로):
  > 예제 3- catch 블록은 사실상 죽은 코드다. Promise의 에러코드는 사실상 try/catch 블록이 콜스택에서 pop된 후에 catch() 체인이 잡기 때문이다.
- **다듬은 정리**: 바깥 `try/catch`와 Promise의 `.catch()`는 **잡는 대상이 다른 두 채널**이다. Promise 거부는 try 블록이 콜스택에서 pop된 뒤 마이크로태스크에서 처리되므로 `.catch()` 채널만 잡고, 바깥 catch는 이 거부에 대해선 실행되지 않는다(죽은 코드).
- 맥락: `try { Promise.reject(e).catch(log) } catch(e){ log }` — `.catch`만 찍히고 바깥 catch는 안 찍힘.
- **이해도 점검**: 이 시나리오에선 정확함. 한 가지 조건만 — 바깥 try/catch가 **완전히** 죽은 건 아니다. **try 블록을 실행하는 동안 동기적으로 던져지는 에러**는 여전히 잡는다. 여기선 try 안에서 하는 일이 "Promise 생성 + `.catch` 등록"뿐이라 동기 에러가 안 나서 *이 거부에 한해* 죽은 코드인 것.
- **더 나아가려면**: 이 "두 채널" 관점이 v0 React 코드(`try { fetch().then().catch() } catch(e){}`)의 이중 구조와 정확히 같은 구조라는 걸 코드에서 다시 짚어보기.
