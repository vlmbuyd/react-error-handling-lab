## 오늘 내가 스스로 깨달은 것

> 출발점: "`try` 안에 왜 체이닝 형태의 `.catch`를 두지? `catch(e){}`랑 중복 아닌가?"
> 라는 의심에서 시작해, 직접 실험하며 "중복이 아니라 다른 채널"이라는 결론까지 스스로 도달함.

다룬 코드 (`src/stages/v0/UserProfileV0.tsx`, 첫 번째 `useEffect`):

```js
try {
  fetchUser()
    .then(setUser)
    .catch(setUserError)        // ← 의심의 출발점
    .finally(() => setIsLoadingUser(false));
} catch (e) {
  setUserError(e as Error);     // ← "중복 아니냐"고 의심했던 곳
}
```

### 💡 `.then`은 기다리지 않는다 — A, C, B
- 이끈 질문: "`fetchUser().then().catch().finally()` 이 한 줄은 fetch가 끝날 때까지 기다렸다 넘어갈까, 시작만 시키고 즉시 다음 줄로 빠질까?"
- 실험: `try` 안에 `console.log('A')` → 체이닝(then 안에 `'B'`) → 체이닝 다음 줄 `console.log('C')` 를 넣고 콘솔 순서 관찰
- 내 말 (원문 그대로):
  > A,C,B 순으로 찍히네. 결론은 fetchUser를 실행시킨다음 (해당 함수는 비동기이므로) 그 다음 동작을 동기적?으로 실해하고, fetch가 완료되면 그제서야 then부터 실행하는거지
- 맥락: `.then`이 블록하지 않고 콜백만 예약한 뒤 즉시 C로 넘어간다는 걸 직접 확인.

### 💡 비동기로 throw된 에러는 try/catch가 못 잡는다
- 이끈 질문: "`.catch(setUserError)`를 지우고 바깥 `catch(e)`만 남긴 뒤 `?fail=true`로 실패시키면, 바깥 catch가 일을 넘겨받아 에러를 잡을까?"
- 실험: 30줄 `.catch(setUserError)` 제거 → `fetch('/api/user?fail=true')` → 결과: **Live 데모는 텅 비고, 콘솔에만** `유저를 불러오지 못했어요` (Unhandled Promise Rejection)
- 내 말 (원문 그대로):
  > 1. fetchUser api 호출
  > 2. api 호출이 비동기이므로 체이닝 다음 코드 블록으로 이동 -> catch{} 블록
  > 3. 이 상태에서는 에러가 안뜨므로 catch{} 블록도 패스
  > 4. 결론적으로 에러가 뜨더라도 이는 비동기적으로 throw 하므로 try/catch의 그 어떤 구문도 catch를 잡지 못한다.
- 맥락: 바깥 `catch(e)`는 A~C 구간(동기) 동안만 감시하는 안전망이라, B 시점의 비동기 실패는 못 잡고 콘솔로 샌다.

### 💡 try/catch와 `.catch()`는 중복이 아니라 "다른 채널"
- 이끈 질문: "`try/catch`는 무엇에 묶여 언제까지 감시하고, `.catch()`는 무엇에 묶여 언제 실행되는가? 왜 둘이 중복이 아닌가?"
- 내 말 (원문 그대로):
  > try/catch는 콜스택 메인스레드에 묶여있어. 여기서의 catch 블록은 try 블록이 에러를 던지지 않고 종료되면 함께 종료되어버려.
  >
  > 반면에 지금과 같이 try 블록 내부의 .catch()는 이벤트 루프에 묶여 있어서 콜백으로 언제든 호출될 수 있어. 즉 try/catch 블록의 실행 종료 시점과는 상관 없이 호출된 fetch 함수의 실행 결과에 따라 호출될 수 잇는거지
- 맥락: `.catch`는 Promise 객체에 등록된 콜백(이벤트 루프가 호출), 바깥 `catch`는 콜스택 동기 채널. 처음의 "중복 아니냐"는 의심이 "아예 다른 메커니즘"으로 뒤집힘.
