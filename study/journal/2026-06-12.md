## 오늘 내가 스스로 깨달은 것

### 💡 렌더 중 throw는 try/catch가 못 잡는다 — try 블록은 이미 끝났으니까
- 이끈 질문: "비동기 rejection을 잡던 그 `.catch` 채널 — 이번 렌더 중 throw도 그게 잡아줄까? 예측부터."
- 본 코드 (발췌):
  ```jsx
  // useEffect의 try/catch는 이미 끝난 뒤,
  // 아래 JSX 렌더 도중 user.profile가 null이라 throw
  <UserCard name={user.name} city={user.profile!.address.city} />
  ```
- 내 말 (원문 그대로):
  > 안잡아줄 것 같아. 왜냐하면 null이 반환된 시점에는 이미 try/catch 블록은 종료돼.
- **다듬은 정리**: 비동기 데이터를 받아오는 `useEffect`의 try/catch는 그 시점에 이미 실행이 끝나 콜스택에서 사라진 상태다. 그 뒤 JSX를 렌더하다가 `user.profile`(null) 접근에서 throw가 나면, 그 throw를 받아줄 try/catch가 같은 스택에 더 이상 없다.
- 맥락: `v0/UserProfileV0.tsx` 63줄 렌더 중 throw, 시나리오 2 예측 단계.
- **이해도 점검**: 결론과 핵심 근거("try 블록이 이미 종료") 정확함. 단, 같은 답변에서 `!`를 옵셔널 체이닝(`?.`)처럼 "null이면 undefined를 반환"하는 것으로 오해했었음 → 아래 항목에서 교정됨.
- **더 나아가려면**: 그럼 이 렌더 throw를 *잡을 수 있는* 곳은 어디인가? (→ ErrorBoundary, v1)

### 💡 `!`는 컴파일 타임에 사라진다 — 런타임엔 보호막이 없다 (자가 수정)
- 이끈 질문: "`user.profile!.address.city`의 `!`는 언제 작동해? 런타임, 아니면 그 전?"
- 본 코드 (발췌):
  ```ts
  user.profile!.address.city   // ! 는 TS 컴파일러에게만 말한다 → 런타임엔 없음
  // 콘솔: const profile = null; profile.address
  // → Uncaught TypeError: Cannot read properties of null (reading 'address')
  ```
- 내 말 (원문 그대로):
  > 근데 생각해보니 ts는 런타임 시점에는 없어지니까 타입 불일치는 컴파일타임에만 잡아줄텐데, .를 통해 접근해서 null이 떠도 그냥 렌더링할 수 있는거 아닌가? 왜 에러를 뱉으면서 화면 자체가 흰화면이 떠버리는거지?
- **다듬은 정리**: TypeScript는 컴파일 후 erase되므로 타입 검사는 컴파일 타임에만 일어난다. `!`(non-null assertion)도 런타임 동작이 전혀 없는 컴파일러용 표식일 뿐이라, 런타임엔 맨손으로 `null.address`에 접근하는 것과 같다. 그래서 `undefined`가 조용히 나오는 게 아니라 `TypeError`가 던져진다.
- 맥락: 콘솔에서 `null.address` 직접 실행해 TypeError 확인.
- **이해도 점검**: "TS는 런타임에 사라진다 → 타입 불일치는 컴파일 타임에만"을 스스로 짚은 게 핵심. 한 가지 교정됨 — `!`는 `?.`와 달리 "null이면 undefined 반환" 같은 런타임 가드가 **아니다**. `!`는 컴파일러 입막음, `?.`가 런타임 단락(short-circuit). 둘을 분리하면 완벽.
- **더 나아가려면**: 만약 `?.`였다면(`user.profile?.address.city`) 이 시나리오에서 throw가 났을까, 아니면 흰 화면을 피했을까?

### 💡 렌더 throw는 React 내부 JS 함수까지 거슬러 올라간다 — 잡을 try/catch가 있다면 React 코드 안에 있어야
- 이끈 질문: "이 throw를 받아낼 try/catch가 있다면, 그건 우리 코드 안일까 React 내부 함수 안일까?"
- 본 코드 (발췌):
  ```
  UserProfileV0          ← 63줄 TypeError throw
    renderWithHooks      ← react-dom 내부 JS 함수
      beginWork
        workLoopSync
          performWorkOnRoot   ← 루트
  ```
- 내 말 (원문 그대로):
  > try/catch는 이미 사라진 시점이니까 적어도 try/catch가 아닌 다른 것이 있겠지? 그래서 에러를 잡아줄 수가 없겠지?
  > React 내부 함수들 중 하나겠지? App.ts의 루트까지 올라갔는데 이 에러를 받아줄 코드가 없다면
- **다듬은 정리**: 우리 컴포넌트(`UserProfileV0`)를 실제로 호출하는 건 React 내부의 JS 함수들(`renderWithHooks`, `beginWork`, `workLoopSync`...)이다. 렌더 중 throw는 이 콜스택을 거슬러 올라가고, 우리가 짠 try/catch는 거기 없으므로 잡으려면 React 내부 함수 중 하나가 try/catch를 가지고 있어야 한다. 루트까지 올라갔는데 받아줄 코드(ErrorBoundary)가 없으면 아무도 못 잡는다.
- 맥락: 실제 콘솔 콜스택이 손으로 그린 스택 그림과 그대로 일치함을 확인(`react-dom_client.js`의 `renderWithHooks`/`beginWork`/`workLoopSync`).
- **이해도 점검**: "React도 그냥 JS 함수 덩어리이고, 그 함수가 내 컴포넌트를 호출한다"를 스스로 정확히 짚음. 실제 콜스택으로 검증까지 됨.
- **더 나아가려면**: React 메시지 "Consider adding an error boundary to your tree" — 그 error boundary는 이 스택 어디에 끼어들어 throw를 가로채는 걸까? (→ v1)

### 💡 React가 트리 전체를 들어내는 이유 — 반쪽짜리 UI는 거짓말을 한다
- 이끈 질문: "React는 왜 '깨진 채로 일부 보여주기'보다 '아예 아무것도 안 보여주기'를 택했을까? 반쯤 망가진 UI가 왜 위험할까?"
- 본 코드/상황 (발췌):
  ```
  송금 화면 렌더 도중:
    "잔액 1,000,000원" 그려짐 → "받는 사람 홍길동" 그려짐
    → 수수료 계산 중 throw → "거래가 취소되었습니다" (못 그려짐)
  ```
- 내 말 (원문 그대로):
  > 비정상적인 값을 받았을 때 어떻게 처리해야할지에 대한 명령을 React 코드가 받지 못했기 때문이 아닐까? 임의로 판단해서 망가진 부분을 제외하고 어떻게 매꿔서 보여주기보다는 그냥 아예 흰화면으로 아무것도 보여주지 말자..를 선택한건가?
  > 3,4번이 오류로 보여지지 않고 1,2번만 보여진다고 했을 때 내가 한 액션이 제대로 수행된 것인지 혼동이 올 것 같아 그래서 혼동을 유발할 바에는 아예 보여주지 말자를 선택한 것 같고. 나름 합리적인 선택인 것 같아
- **다듬은 정리**: React는 에러 난 컴포넌트를 어떻게 복구해야 할지에 대한 명령(= ErrorBoundary)을 받지 못하면, 멋대로 추측해 반쪽을 때워 보여주지 않고 트리 전체를 들어낸다. 일부만 그려진 화면은 사용자에게 "내 액션이 정상 수행됐다"는 잘못된 확신을 줄 수 있어, 그 혼동을 유발하느니 차라리 빈 화면을 보여주는 게 안전하다는 합리적 선택이다.
- 맥락: 시나리오 2 실행 시 `UserProfileV0` 하나의 에러로 페이지 전체가 흰 화면이 된 것을 관찰한 뒤.
- **이해도 점검**: 정확함. "React는 임의로 매꾸지 않는다" + "반쪽 UI는 거짓말을 해서 빈 화면보다 위험하다"는 React 기본값의 설계 의도를 스스로 도달함. 이 흰 화면이 바로 ErrorBoundary가 존재하는 이유(= '대신 보여줄 것'을 명령으로 주는 장치)로 이어짐.
- **더 나아가려면**: ErrorBoundary를 주면 React는 트리 전체 대신 무엇을 보여줄까? "전부 아니면 전무(all-or-nothing)"를 어떻게 쪼갤까? (→ v1)
