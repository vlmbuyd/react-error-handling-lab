# 2026-06-17 — v2: 에러 state를 ErrorBoundary로 옮겨 v1의 throw 트릭을 없애다

- 다룬 단계: v2
- 다룬 시나리오: 비동기 에러 경로(`.catch(showBoundary)`) / 렌더 throw 경로(`getDerivedStateFromError`) / [버티는 줄] 실험(Provider·setState 제거)

> v1과 `UserProfileV2`를 나란히 비교하며 시작했다. "사라진 두 줄(`useState` + `if throw`)이
> 어디로 갔나"를 따라가다, 에러 state가 자식에서 ErrorBoundary로 옮겨간 것이 v2의 핵심임을
> 자기 말로 도달했다. 두 에러 경로(동기 throw / 비동기 showBoundary)가 하나의 `this.state.error`로
> 합류한다는 것까지 정리. 핵심 질문 1번의 v2 확장 완료.

## 오늘 내가 스스로 깨달은 것

### showBoundary는 ErrorBoundary가 만들고, useErrorBoundary는 그걸 꺼내는 통로일 뿐

- 이끈 질문: "`showBoundary`는 어디서 만들어져요? `useErrorBoundary.ts`가 만들어요, 어디서 가져와요?"
- 본 코드:
  ```jsx
  showBoundary = (error: Error) => this.setState({ error });   // ErrorBoundaryV2
  <BoundaryContext.Provider value={{ showBoundary: this.showBoundary }}>
  ```
- 내 말 (원문 그대로):
  > showBoundary는 에러바운더리 클래스 내부에서 만들어. useBoundary는 해당 컨텍스트 내부에서 showBoundary를 가져올 수 있는 훅일뿐이야
- **다듬은 정리**: `showBoundary`는 `ErrorBoundaryV2` 안에서 정의돼 Context로 자식에게 내려간다. `useErrorBoundary`는 그 Context에서 `showBoundary`를 꺼내오는 통로일 뿐, 직접 만들지 않는다.
- 맥락: `ErrorBoundaryV2.tsx` 31·40줄, `useErrorBoundary.ts`, `context.ts`.
- **점검**: 정확함.
- **다음 한 걸음**: 그 `showBoundary` 몸통 안엔 뭐가 있나? (→ setState)

### 에러 state가 자식에서 ErrorBoundary로 옮겨갔다 — 단일 소스

- 이끈 질문: "v2에선 그 state가 누구한테로 옮겨갔죠? showBoundary 몸통 안엔 뭐가 있어요?"
- 본 코드:
  ```jsx
  state = { error: null };
  showBoundary = (error) => this.setState({ error });
  ```
- 내 말 (원문 그대로):
  > v2에서는 에러 state가 에러바운더리 클래스 내부 상태 단일 소스로 관리하게 됐어.
  > showBouondary 내부에 setState가 있고, 아마 이 setState는 에러 상태를 업데이트해주는 setState함수겠지?
- **다듬은 정리**: v1에선 에러 state가 자식 컴포넌트 안에 있었지만, v2에선 ErrorBoundary가 `this.state.error` 하나로 관리한다. `showBoundary`는 그 state를 갱신하는 `setState` 호출이다.
- 맥락: `ErrorBoundaryV2.tsx` 18·31줄.
- **점검**: 정확함. (사실 보강: `this.setState`는 `useState`의 함수형 버전이 아니라 원조다. `extends Component`가 물려준 메서드라 클래스 state여도 똑같이 재렌더를 트리거한다.)
- **다음 한 걸음**: state만 바꾸고 throw는 안 하는데 왜 fallback이 뜨나?

### 왜 v2는 throw가 필요 없나 — state가 이미 ErrorBoundary 안에 있으니 건너보낼 게 없다

- 이끈 질문: "v1은 왜 throw라는 '건너뛰기'가 필요했고, v2는 왜 그게 통째로 필요 없어졌나?"
- 본 코드:
  ```jsx
  render() {
    if (this.state.error) return <Fallback ... />;   // 34줄: 같은 state를 분기
  ```
- 내 말 (원문 그대로):
  > 에러 state는 에러바운더리 클래스 내부에 존재해. 즉 catch(showBoundary)로 에러가 전달됐을 때 showBoundary가 실행된 후 에러 state를 업데이트하고, render 메서드의 if 절이 실행되어서 Fallback ui가 보여져.
  > v1에서는 컴포넌트 자체적으로 에러 상태를 관리하고 있었으므로 부모인 에러바운더리 클래스에 에러를 전달해줄 방법이 필요했떤 거고, 그 방법이 setState로 렌더링을 트리거한 후 throw를 통해 던져준다 였어.
  > 근데, v2에서는 클래스 자체적으로 에러 상태를 관리하고 그에 따라 Fallback을 렌더링하므로 이 전달과정이 생략될 수 있었던거지. 그리고 useContext를 활용한 훅을 통해서 자식 어디서나 접근할 수 있고
- **다듬은 정리**: v1은 에러 state가 자식에게 있어 부모로 "전달"하려면 render 중 throw가 필요했다. v2는 에러 state를 ErrorBoundary가 직접 들고 있어 전달할 게 없다. `setState`로 ErrorBoundary가 재렌더되고 자기 `render()`의 if가 fallback을 띄운다.
- 맥락: `ErrorBoundaryV2.tsx` 34줄, v1 12줄 트릭과 대비.
- **점검**: 정확함. v1↔v2의 차이를 "state가 누구한테 있나"로 정확히 짚음.
- **다음 한 걸음**: 그럼 동기(렌더) throw는 v2에서 누가 잡나? componentDidCatch는 죽은 코드인가?

### 두 에러 경로가 하나의 this.state.error로 합류한다 (getDerivedStateFromError vs componentDidCatch 재확정)

- 이끈 질문: "렌더 throw에서 실제로 fallback을 띄우는 건 getDerivedStateFromError? componentDidCatch?"
- 본 코드:
  ```jsx
  static getDerivedStateFromError(error) { return { error }; }       // 20줄: state 채움
  componentDidCatch(error, info) { this.props.onError?.(error, info); } // 24줄: 로깅만
  ```
- 내 말 (원문 그대로):
  > 컴포넌트가 에러를 throw하면 getDerivedStateFromError를 호출해서 에러 상태를 업데이트해
  > componentDidCatch는 이와 별개로 커밋 페이즈에서 주로 로깅하는 용도로 호출되고.
  > 결과적으로 렌더 상황에서 에러를 throw하면 여전히 getDerivedStateFromError가 잡는 것이 맞고, 비동기 상황 (e.g. .catch())에서 지금 상황에서는 showBoundary를 통해 에러를 잡고 업데이트하는거고
- **다듬은 정리**: 렌더 throw(동기)는 `getDerivedStateFromError`가 state를 채워 fallback을 띄운다. 비동기 에러는 `showBoundary`의 `setState`가 같은 state를 채운다. 두 경로가 하나의 `this.state.error`로 모여 34줄 if 하나가 둘 다 처리한다. `componentDidCatch`는 옆에서 `onError` 콜백으로 로깅만 한다(state 안 건드림).
- 맥락: `ErrorBoundaryV2.tsx` 20·24·34줄. (중간에 한 번 componentDidCatch가 "잡는다"로 헷갈렸다가 코드 몸통을 보고 스스로 교정함.)
- **점검**: 교정 후 정확함. `onError`는 부모가 넘기는 선택적 로깅 콜백(Sentry 등)이라 지워도 fallback은 뜨고 로깅만 사라진다.
- **다음 한 걸음**: (없음 — 정리됨)

## 한눈에 복습 (내 말로 다시 설명해보기)

- **v1→v2 핵심** — 에러 state가 ***에서 ***로 옮겨갔다 (그래서 throw가 필요 없어졌다)
- **showBoundary** — 누가 만들고(클래스) 누가 꺼내나(useErrorBoundary=통로) / 몸통은 `this.setState({error})`
- **왜 throw가 사라졌나** — 전달할 곳에 state가 이미 있으니 \_\_\_ 과정이 생략됨
- **두 경로 합류** — 동기 throw → \_\_\_ / 비동기 → showBoundary / 둘 다 `this.state.error` 한 곳
- **componentDidCatch** — state 안 건드림, `onError`로 \_\_\_만
- **[버티는 줄]** — 40줄 Provider(통로) + 31줄 setState(주입). 지우면 v1 12줄 지웠을 때처럼 영원한 로딩으로 회귀
- **v2 diff 한 문장(완성)**: "setState→throw 보일러플레이트를, ErrorBoundary가 에러 state를 단일 소스로 들고 Context로 showBoundary를 내려보냄으로써 없앴다"

# 2026-06-17 (이어서) — v3: 로딩도 throw다 (Suspense의 심장)

- 다룬 단계: v3 (throw-promise / render-as-you-fetch)
- 다룬 코드: `createResource.ts`, `UserProfileV3.tsx`

> v0의 삼항 분기가 사라진 컴포넌트에서 출발해, "로딩을 왜 throw하나"를 따라갔다.
> Suspense가 로딩을, ErrorBoundary가 에러를 받는다는 짝을 맞추고, status가 클로저 변수라는 것,
> 던지는 게 Promise여야 하는 이유, fetch를 컴포넌트 바깥에 둔 이유와 그 한계까지 자기 말로 도달했다.

## 오늘 내가 스스로 깨달은 것

### status는 React가 주입하는 게 아니라 createResource의 클로저 변수다

- 이끈 질문: "이 `status`는 누가 선언하고, 누가 값을 바꾸나요? React가 주입하는 게 맞아요?"
- 본 코드:
  ```ts
  let status: 'pending' | 'success' | 'error' = 'pending';
  const promise = promiseFn().then(
    (data) => { status = 'success'; result = data; },
    (error) => { status = 'error'; result = error; },
  );
  ```
- 내 말 (원문 그대로):
  > createResource가 변수로 선언하고 PromiseFn 콜백 실행 시에 값을 바꾸지
- **다듬은 정리**: `status`/`result`는 `createResource` 안에서 선언된 클로저 변수다. React가 주입하지 않는다. fetch가 끝나면 `then` 콜백이 그 값을 바꿔놓고, `read()`는 그 값을 들여다본다.
- 맥락: `createResource.ts` 11·15~18·25줄.
- **점검**: 정확함. "React가 자동 주입하나"라는 처음 짐작을 코드를 보고 스스로 교정함.
- **다음 한 걸음**: 그 클로저 변수는 컴포넌트가 재렌더돼도 그대로 살아있나?

### 던지는 게 Promise여야 하는 이유 — Suspense가 "끝나면 깨워줘" 알람으로 쓴다

- 이끈 질문: "왜 하필 Promise를 던질까? `throw 'loading'`이면 Suspense가 언제 다시 시도할지 어떻게 알죠?"
- 본 코드:
  ```ts
  case 'pending': throw promise;  // Suspense가 잡음
  ```
- 내 말 (원문 그대로):
  > then을 통해 resolve되면 성공한 시점을 알 수 있다는거?
  > 서스펜스가 여기서 리렌더링한다면 모듈 스코프의 변수는 초기화되지는 않겠찌?
- **다듬은 정리**: Suspense는 던져진 Promise에 콜백을 붙여, resolve되는 순간(=fetch 끝난 시점)을 알아채고 자식을 다시 렌더한다. 문자열로는 그 타이밍을 알 수 없다. 재렌더 때 모듈 스코프 변수는 초기화되지 않으므로, 다시 불린 `read()`는 이미 `success`가 된 값을 그대로 돌려준다.
- 맥락: `createResource.ts` 26줄, `UserProfileV3.tsx` 11줄(모듈 스코프).
- **점검**: 정확함. Promise를 "데이터 운반"이 아니라 "재시도 신호"로 본 것이 핵심.
- **다음 한 걸음**: 그래서 fetch를 컴포넌트 바깥에 둔 이유는?

### v3가 짧은 이유 — 상태 관리와 로딩/에러 UI를 컴포넌트 밖으로 빼서 컴포넌트는 성공만 본다

- 이끈 질문: "UserProfileV3는 어떻게 성공한 경우만 신경 쓰고 로딩·에러를 안 적을 수 있게 됐나?"
- 본 코드:
  ```jsx
  const user = userResource.read();   // 로딩이면 throw promise, 에러면 throw error, 성공이면 값
  ```
- 내 말 (원문 그대로):
  > 데이터 패칭 상태와 데이터 자체 상태를 훅처럼 createResource에서 관리해주고 그에 따른 UI 렌더링도 컴포넌트 외부에서 처리해주니까 컴포넌트는 성공 케이스만 집중할 수 있어서 코드가 깔끔해지는 장점이 있을 것 같아.
- **다듬은 정리**: 패칭 상태와 데이터를 `createResource`가 관리하고, 로딩·에러 화면은 바깥 Suspense와 ErrorBoundary가 맡는다. 그래서 컴포넌트는 성공 케이스만 적으면 되고 분기가 사라진다.
- 맥락: `UserProfileV3.tsx` 15줄, v0 삼항 분기와 대비. (= v3 diff 한 문장)
- **점검**: 정확함. v3의 핵심 이득을 자기 말로 한 문장에 담음.
- **다음 한 걸음**: 그런데 fetch를 바깥에 두면 무엇을 못 하게 되나?

### fetch를 바깥에 둔 이유와 그 한계 — props/동적 id를 못 본다

- 이끈 질문: "props로 받은 userId를, 컴포넌트 렌더 전에 실행되는 모듈 스코프 11줄이 가져올 수 있나? createResource를 컴포넌트 안으로 옮기면?"
- 본 코드:
  ```jsx
  const userResource = createResource(() => fetchUser());  // 컴포넌트 함수 바깥, import 때 1번
  ```
- 내 말 (원문 그대로):
  > id와 같은 파라미터를 api 경로에 포함해야 하는 경우는 컴포넌트가 호출된 시점 이후니까 지금과 같은 패턴과는 맞지 않네.
  > 컴포넌트 내부에서 호출하게 되면 컴포넌트가 호출될 때마다 status 값이 pending 상태로 초기화되면서 다시 불러올 것 같아. 반대로 ... 외부에 두게되면 성공한 status 값과 데이터는 클로저 변수에 저장이 되고, 컴포넌트가 재호출되더라도 로딩 시간 없이 바로 받아올 수 있는건가?
- **다듬은 정리**: 모듈 스코프 fetch는 import 시점에 한 번 발사돼 컴포넌트 렌더보다 앞선다. 그래서 미리 받아두는 이득이 있지만, props 같은 런타임 값(동적 id)은 볼 수 없다. 반대로 컴포넌트 안에서 만들면 props는 보지만 재렌더마다 새 pending이 만들어져 끝나지 않는다.
- 맥락: `UserProfileV3.tsx` 11줄, inside/outside 트레이드오프.
- **점검**: 정확함. "render 바깥=지속성 있지만 props 못 봄 / render 안=props 보지만 무한 suspend"라는 v3의 한계를 정확히 짚음. 이게 v4·Suspensive로 넘어가는 동력.
- **다음 한 걸음**: props로 만든 키로 상자를 캐시해 둘 수 있다면? (→ v4 입구)
