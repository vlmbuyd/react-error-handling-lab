# /commit — 날짜별 커밋 + 푸시

목표: 작업 디렉토리의 변경사항을, **일지처럼 날짜가 있는 기록은 그 날짜로**,
나머지는 의미 단위로 나눠 커밋하고 푸시한다. **git commit을 실행하는 모든 순간,
그 직전에 반드시 사용자 컨펌을 받는다** — 이게 가장 중요한 규칙이다.

## 0. 절대 규칙

- **컨펌 없이 `git commit`을 실행하지 않는다.** 커밋이 1개든 여러 개든, 분할 계획과
  각 커밋의 메시지를 먼저 텍스트로 보여주고 사용자가 명시적으로 진행을 승인한 뒤에만 커밋한다.
- **push도 같은 컨펌에 포함**하되, 사용자가 커밋만 원하는지 푸시까지 원하는지 애매하면 물어본다.
- `--force`, `--no-verify`, `--amend`, `reset --hard` 등 위험한 옵션은 쓰지 않는다.
- 의심스러운 파일(실험용으로 임시로 바꿔둔 코드, 스크래치 파일 등)은 마음대로 포함/제외를
  판단하지 않고 `AskUserQuestion`으로 물어본다 (추천 옵션 포함).

## 1. 변경사항 파악 및 분류

`git status`, `git diff`로 모든 변경/untracked 파일을 본 뒤 세 그룹으로 나눈다.

1. **날짜 있는 기록** — `study/journal/YYYY-MM-DD.md` 같이 파일명/내용에 날짜가
   박혀있는 새 파일. 이 날짜가 그대로 그 커밋의 날짜가 된다.
2. **날짜 없는 공유 파일** — `study/PROGRESS.md`, `study/journal/README.md`,
   `.claude/skills/**/SKILL.md`, `CLAUDE.md` 등 여러 세션에 걸쳐 누적 수정되는 파일.
   이 파일들의 diff를 어느 날짜 커밋에 합칠지는 **내용으로 추론**한다:
   - `PROGRESS.md`의 "마지막 세션 날짜" 필드가 최종적으로 가리키는 날짜, 그리고
     일지 목록에 새로 추가된 항목들의 날짜를 보고 그 변경이 어느 세션에서 일어났는지 가늠한다.
   - 여러 세션의 변경이 한 diff에 뭉쳐 있으면(예: 일지 3개를 한 번에 작성한 경우),
     각 날짜 세션이 끝난 시점의 `PROGRESS.md` 상태를 **그 날짜 일지의 "다음에 이어갈 질문"/
     "다음 세션 시작점" 섹션 내용을 근거로 역산**해 중간 상태를 재구성한다. 완벽히 복원은
     안 되더라도, 합리적인 추정으로 그 날짜 시점 상태를 만든다.
   - `README.md`/`SKILL.md`처럼 특정 세션에서 새로 생긴 가이드 문구는, 그 문구가 예시로
     드는 날짜나 그 세션 일지의 내용과 맞춰 어느 날짜 커밋에 속하는지 판단한다.
   - 도저히 못 가르면 가장 마지막(최근) 날짜 커밋에 합치고, 그렇게 처리했음을 사용자에게
     알린다 — 추측한 것처럼 숨기지 않는다.
3. **그 외 변경** — 소스 코드(`src/**`) 실험용 변경, 스크래치 파일 등 학습 기록과
   무관해 보이는 변경. 날짜 커밋에 끼워 넣지 말고, 포함할지/제외할지 사용자에게 먼저 물어본다.
   (과거 사례: `fetchers.ts`에 `?fail=true`를 하드코딩해둔 실험 상태, `src/prac.js` 같은
   콘솔 스크래치 파일 — 둘 다 제외하고 작업 디렉토리에 그대로 둔 전례가 있음.)

## 2. 커밋 계획 세우기

날짜 오름차순으로 커밋 목록을 만든다. 각 항목마다:
- 포함할 파일
- 커밋 메시지 (제목 1줄 + 본문 1~3줄, 기존 커밋 스타일 `docs: ...` 따라가기 — `git log`로 확인)
- 그 날짜로 backdate할 `GIT_AUTHOR_DATE`/`GIT_COMMITTER_DATE` (예: `YYYY-MM-DDT22:00:00`,
  같은 날 기존 커밋이 있으면 그보다 늦은 시각으로)

## 3. 컨펌 (필수, 건너뛰지 않음)

계획 전체를 사용자에게 보여준다 — 몇 개의 커밋으로 나눴는지, 각 커밋에 어떤 파일이
들어가는지, 커밋 메시지 초안, 추론/추측이 들어간 부분(중간 상태 재구성, 날짜 배정 등)은
**추측임을 명시**, 제외한 파일과 그 이유. 사용자가 수정 요청하면 반영 후 다시 보여준다.
**사용자가 명확히 승인("ㄱㄱ", "ㅇㅇ", "진행해" 등)한 뒤에만** 다음 단계로 넘어간다.

## 4. 실행

승인받은 순서대로, 각 커밋마다:

```bash
GIT_AUTHOR_DATE="YYYY-MM-DDT22:00:00" GIT_COMMITTER_DATE="YYYY-MM-DDT22:00:00" \
  git add <해당 파일들> && \
GIT_AUTHOR_DATE="YYYY-MM-DDT22:00:00" GIT_COMMITTER_DATE="YYYY-MM-DDT22:00:00" \
  git commit -m "$(cat <<'EOF'
<제목>

<본문>

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
EOF
)"
```

날짜 없는 일반 변경(분류 3, 사용자가 포함하기로 한 것)은 backdate 없이 현재 시각으로
별도 커밋한다.

전부 커밋한 뒤 `git push`로 푸시한다 (3단계 컨펌에 push도 포함했다면 별도로 다시 묻지 않음).
마지막에 `git log --oneline -N`으로 결과를 보여준다.

## 하지 말 것

- 컨펌 받기 전에 `git commit` 또는 `git push` 실행.
- 추측으로 채운 부분을 마치 원본 그대로인 것처럼 말하기 — 추론했다는 사실은 항상 밝힌다.
- 애매한 파일(실험 코드, 스크래치 파일)을 묻지 않고 임의로 포함시키거나 빼기.
- 여러 날짜의 변경을 뭉쳐서 하나의 커밋으로 만들기(분리 가능한데 귀찮아서 합치는 것 금지).
